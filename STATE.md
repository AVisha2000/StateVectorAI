## High Priority

- Boundary-safe synthetic sampling remains the top correctness gate. Generated datasets are flattened, `qllm/data/text.py::sample_batch` is shared by train/eval, and `qllm/data/datasets.py::DATASET_KINDS` is stale. Next: promote one L2 single-fix run to add dataset boundary metadata, boundary-aware split/sampling, cache metadata, and focused tests before trusting new synthetic-memory results.
- Comprehensive config validation is the next critical correctness gate. `.venv` validation showed all checked-in configs pass, but intentionally bad configs for data kind, architecture, embedding/head type, two-stream condition, train sizes, `seq_len > max_seq_len`, backend, ansatz, and readout currently return no validator errors. Next: after boundary sampling, promote an L2 validator pass that centralizes registries and makes CLI/dashboard queue paths share the same pre-run validation.
- Two-stream metric honesty is the third critical correctness gate. `qllm/models/two_stream.py` pools the full token window for quantum/classical sentence encoders, while `tests/test_two_stream.py` only proves causality for `encoder_kind="none"` and dashboard result views still label these runs with generic validation perplexity. Next: choose either causal prefix pooling or explicit side-information labeling across benchmark output, dashboard comparisons/results, docs, and RESULTS wording before ranking two-stream runs with strict autoregressive LMs.
- Dashboard full fairness-field comparison should land before study statistics are trusted. `qllm/dashboard/lab.py::fairness_flags` now checks core training/preprocessing fields, but it does not compare synthetic generator/task fields such as `data.gen_seed`, `data.gen_qubits`, `data.gen_theta_x`, contextual fields, or cancellation density; a temp-DB probe showed a candidate/baseline pair differing only in `data.gen_seed` still reports `same_preprocessing=True` and `single-run candidate better`. Next: promote a later L2 fix to define per-family intentional-difference allowlists, compare all non-allowlisted flattened config fields, expose mismatched keys in the API/UI, and add dashboard tests proving synthetic-generator mismatch fails fairness.
- Parameter-matched analogue generation is the next dashboard evidence gate. `qllm/models/model.py::matched_classical_d_ff` exists and `benchmarks/ablation.py` uses it, but `qllm/dashboard/analogues.py` and preset twins mostly do component swaps or curated twins. A parameter-count probe showed `quantum-ffn-4q -> classical-small` has a parameter delta ratio about `-0.578` at vocab 32 while the matched FFN helper would choose `d_ff=4`; `qrnn-small -> gru-small` is also far from matched at small recurrent vocabularies. Next: add per-family analogue builders or ladders that record analogue type, parameter delta ratio, and known limitations before evidence/study cards can treat comparisons as parameter-matched.
- Kernel advantage split hygiene is the next evidence-hardening item after dashboard fairness and analogue matching. `qllm/quantum/advantage.py::kernel_ridge_r2` selects the best ridge regularization directly on the test split, and `benchmarks/advantage_probe.py` reports only test R2 values without selected regularization or split sizes. Next: promote an L2 fix to add deterministic train/validation/test selection or nested CV, return selected regularization metadata, and keep the positive/negative controls passing.
- Study-level paired statistics are the next dashboard evidence-hardening item after full fairness fields and analogue matching are enforced. `qllm/research_protocol.py::paired_stats` and `classify_claim` already support confidence intervals, p-values, effect size, win rate, and cautious claim levels, but `qllm/dashboard/studies.py` currently labels studies from wins plus mean/std `delta_val_ppl`, and the Study/Study Report UI surfaces no p-value, confidence interval, effect size, or protocol claim level. Next: wire fair completed study pairs into `paired_stats`/`classify_claim`, surface the statistics in the study detail/report markdown, and add dashboard tests for weak positive but non-significant multi-seed effects.
- Result claim ledger is the next claim-safety item for dashboard highlights. There is no `results/claims.json` or `docs/CLAIMS.md`; `RESULTS.md` stores claim status only in prose; and `qllm/dashboard/explore.py::result_dashboard_payload` summary cards can highlight champion or promising rows where `claim_level` is `None` for offline `runs` rows. Next: define a small ledger schema with claim ID, protocol claim level, supporting run/job IDs, limitations, next decisive check, and status; mirror it into result summary payloads; and add tests proving every highlighted result has a protocol claim level before any advantage-like wording appears.
- Dashboard comparison warnings need to become first-class API data before weak comparison results are easy to read safely. `qllm/dashboard/lab.py::comparison_research_payload` currently returns `fairness`, `verdict`, `resource_normalized`, and `evidence_ladder`, but no structured `warnings` or `caveats` array; `qllm/dashboard/evidence.py::comparison_evidence_ladder` can still label a single fair anecdote as `promising run` while multi-seed and ablation/control rungs remain false. Next: add a backend warning builder for one-seed, unmatched parameters, missing frozen/random controls, two-stream side-information, slow negligible gains, and synthetic config mismatch; expose it in comparison payloads and render warning panels in Launch/Comparison/Studies with dashboard tests.
- Dashboard dataset import provenance is only partially implemented. `qllm/dashboard/datasets.py` records source, split, text column, corpus path, row count, char count, preview, and `qllm/resultsdb.py` stores a `ts` timestamp, with tests covering empty source and invalid row limits; however the schema/payload lacks source revision, requested row limit, corpus hash, truncation state/warnings, and any byte/character cap. A mocked import with three rows and `row_limit=2` stored only two rows with no truncation flag or warning, and `Datasets.jsx` displays rows/chars/source but not timestamp, hash, row-limit, revision, or truncation provenance. Next: add a schema/payload migration for import provenance, compute a corpus hash, record row/size caps plus truncation warnings, render those fields in the Datasets page, and add focused tests for reproducibility metadata.
- Quantum compile/runtime telemetry is the next GPU-scaling observability gap. `qllm/dashboard/resources.py::quantum_resource_estimate` already computes static queue-time `state_dim`, token calls, component multiplier, score, and band, and `qllm/research_protocol.py::resource_ledger_from_config` exposes a static ledger tested in `tests/test_research_protocol.py`; however `qllm/train/loop.py::fit` only prints first-step JIT time and persists total `wall_seconds`, saved `summary.json` files lack compile/steady-step/peak-memory fields, and the `runs` table has only `wall_seconds` plus `config_json`. Dashboard result rows show wall/qubits/depth/backend/resource band and sort by validation perplexity, not by wall time or resource cost. Next: promote an L2 telemetry pass to measure first-step compile time, steady-state step time, optional peak memory, and resource ledger fields; persist them in summaries/DB/config metadata; and surface sortable resource-cost columns in dashboard result/GPU panels with tests.
- Durable dashboard queue recovery is a high-priority reliability item before unattended or long dashboard runs. `qllm/dashboard/runner.py::ExperimentQueue` stores pending job IDs in a process-local `queue.Queue`, does not reload `lab_jobs` on startup, and `lab_jobs` has no worker ID, heartbeat, claim timestamp, or stale-running recovery state. The current DB has no queued jobs, but job `42` and its `live_runs` row have been `running` since 2026-06-21T18:36 with step 0, showing stale-running detection is missing. Next: promote a later L2 dashboard fix to add DB-backed claiming, startup queued-job rehydration, heartbeat/stale recovery, and tests that restart resumes queued jobs without cancelling active work.
- Dashboard local safety guardrails are still missing for any remote-bound dashboard session. `qllm/dashboard/run.py` defaults to `127.0.0.1`, but accepts arbitrary `--host` values with no `--allow-remote` gate; `qllm/dashboard/server.py` uses wildcard CORS; `/api/plot/{name}` builds `RESULTS_DIR / name` without a resolved containment check or 404 errors; and the SPA fallback serves `FRONTEND_DIST / full_path` if it is a file, without containment checks. Next: add explicit remote opt-in, localhost-only CORS by default, resolved-path containment for plots/static fallback, `/api/status` bind-safety warnings, and dashboard tests for CORS plus traversal rejection.
- Checkpoint/resume is the next run-reliability blocker after durable queue recovery and local safety. `qllm/train/loop.py::fit` creates a `TrainState` and optimizer state in memory, but only writes final `params.msgpack` and `summary.json`; it does not save periodic latest/best checkpoints, optimizer state, current step, or RNG position. Dashboard jobs call `fit(..., should_cancel=...)` without a resume path, and `ResultsDB` stores per-step curves plus final run rows but no checkpoint path or resumed-step metadata. Next: promote a later L2 fix to save latest and best-validation checkpoints, restore params/optimizer/RNG/step, expose `resume_from` through CLI/dashboard/benchmark paths, and record resume metadata in summaries plus DB.
- Idempotent per-step logging is the next dashboard curve reliability item. The `steps` table has an autoincrement primary key but no uniqueness constraint on `(run_key, step, name)`, and `qllm/resultsdb.py::log_step` uses plain `INSERT INTO steps`; a temp-DB probe that logged `train_loss` twice for the same run key and step returned two curve rows while `live_runs` only kept the latest scalar. Next: promote a later L2 fix to choose replace-in-place versus new-run-key semantics, enforce it in schema/write paths, and add dashboard tests proving reruns cannot silently double-count curve points.
- Generation contract across architectures is the next model-test compatibility item. `qllm/train/loop.py::generate` assumes `model.cfg.max_seq_len`; a tiny initialized-params probe showed transformer and two-stream generation work, but `GRULM` and `QRNNLM` fail immediately with `AttributeError: ... object has no attribute "cfg"`. Dashboard model testing currently marks prompt generation ready for any completed text job with config and `params.msgpack`, so completed GRU/QRNN text jobs can expose a generate button that crashes instead of returning a clear unsupported-architecture message. Next: promote a later L2 fix to pass `context_len` or architecture metadata explicitly, gate dashboard support by architecture, and add smoke tests for transformer, GRU, QRNN, and two-stream behavior.
- GPU queue remains the main research backlog, but its highest-value experiments depend on synthetic sequence data. Next: defer long GPU runs until the boundary-sampling fix lands, then use `$qllm-experiment-runner` and `$qllm-research-protocol` to pick the smallest decisive run.
- The loop run-count cap is now unlimited, but L1 remains report-only and the 100k token/day budget still applies. Next: continue report-only loops unless intentionally promoting to L2.

- Local loop scaffold and enhancement docs remain untracked. Treat them as expected planning state until the user asks for staging/commit.
- Global Python 3.12 cannot import `yaml`; the repo `.venv` can. Use `.venv\Scripts\python.exe` for validation/test probes unless the shell environment is activated.
- `RESULTS.md` is large and claim-sensitive; route any strengthened conclusions through research-protocol review.
- Two-stream v0.16 wording in `RESULTS.md` is statistically cautious, but it does not yet distinguish teacher-forced side-information perplexity from strict autoregressive LM perplexity.
- The dashboard comparison page shows high-level fairness badges only (`dataset`, `seed`, `steps`, `eval`, `device`, `roles`); even existing `matched_config_fields` details are not surfaced to users yet.
- The model builder currently marks any generated analogue as `paired-ready` once an analogue exists, even though automatic component-swap analogues only say parameter matching will be verified after metrics are recorded.
- The comparison evidence ladder only marks the `parameter_matched` rung when final `n_params` deltas are within tolerance, but the separate `matched_baseline` rung means only that a linked baseline exists.
- Comparison API payloads currently lack a structured warning list, so the UI cannot reliably show the warning set required by `docs/ENHANCEMENT_PLAN.md` item 14.
- Hugging Face import has a hard row-limit guard and basic tests, but no corpus hash, revision, requested-row-limit field, byte/char cap, or truncation warning for dataset reproducibility.
- Static resource estimates exist for queue-time guardrails, but completed runs do not durably record first-step compile time, steady-state step time, peak memory, or a sortable resource-cost ledger.
- Advantage-potential results in `RESULTS.md` are framed as methodology validation, but the reported R2 values predate split-hygiene metadata. Revisit wording only after the kernel-ridge selection fix lands.
- Research-protocol paired-stat primitives are already present and tested in `tests/test_research_protocol.py`; the dashboard study gap is integration and reporting, not a new statistics design.
- Dashboard result rows from linked lab comparisons expose `verdict_label` and `claim_level`, but summary cards only show `verdict_label`, and historical/offline `runs` rows have no protocol claim metadata.
- No dashboard-local-safety tests currently cover wildcard CORS, plot path traversal, SPA fallback containment, or non-loopback bind warnings.
- Benchmark suite scripts already skip completed `(suite, variant, dataset, seed, steps)` result cells through `ResultsDB.exists`, so their "resumable" wording means whole-cell skip/retry, not mid-run optimizer-state recovery.
- Dashboard curve readers in `qllm/dashboard/queries.py` and `qllm/dashboard/workspace.py` group whatever `fetch_steps` returns; `fetch_steps` orders by step only, so duplicate rows for the same metric/step can render as repeated points rather than a clean replacement.
- Existing generation tests cover only the transformer path in `tests/test_integration.py`; recurrent and two-stream tests verify logits/dispatch but not `generate()`, and dashboard tests only verify artifact capability reporting, not manual generation execution across architectures.

## Handoff

- Decide whether to promote the boundary-safe synthetic sampling item to L2. If yes, keep it to one narrowly scoped fix and run focused data/training tests before proposing completion.
- After boundary sampling and config validation, decide the two-stream policy: causalize the sentence stream or mark two-stream metrics as side-information everywhere they appear.
- For dashboard fairness expansion, decide the intentional-difference allowlist per comparison family so model-only changes pass while synthetic data/preprocessing drift fails loudly.
- For analogue generation, decide the first family to fix: quantum FFN can reuse `matched_classical_d_ff`, while QRNN/contextual likely need GRU ladders rather than a single arbitrary hidden size.
- For kernel advantage screens, decide whether the first L2 fix should return a richer score object while keeping the existing `kernel_ridge_r2` scalar wrapper for compatibility.
- For study evidence, decide whether the first dashboard L2 pass should make `classify_claim` the source of the study verdict once at least three fair pairs exist.
- For the result claim ledger, decide whether the durable source should be `docs/CLAIMS.md` for human review, `results/claims.json` for dashboard loading, or both generated from one schema.
- For dashboard comparison warnings, decide the warning severity vocabulary and whether `promising run` should be renamed or demoted when warnings include one-seed, missing-control, parameter, side-information, or cost caveats.
- For dataset import provenance, decide whether existing `lab_datasets` rows should be migrated in place with nullable columns or versioned through a new metadata JSON field, and whether corpus hashes should cover normalized joined text or raw source rows.
- For runtime telemetry, decide whether new fields belong as explicit `runs` columns, `config_json` metadata, or a separate resource ledger table, and whether peak memory should use JAX backend APIs opportunistically with `None` fallback.
- For durable queue recovery, decide whether stale `running` rows should become `error`, `recoverable`, or be re-claimed when their worker heartbeat expires.
- For dashboard local safety, decide whether remote exposure should be a CLI flag, env flag, or both, and what localhost origins should be allowed for Vite development.
- For checkpoint/resume, decide checkpoint cadence, retention policy, and whether cancelled/interrupted dashboard jobs should resume automatically, require a user action, or remain terminal while exposing a manual resume button.
- For per-step logging idempotence, decide whether reusing a `run_key` should replace prior curve points, delete the old curve at `start_run`, or generate a distinct attempt/run key so retries remain auditable.
- For generation support, decide whether GRU/QRNN should become first-class generation targets via explicit `context_len`, or whether dashboard/API should intentionally return unsupported for some recurrent architectures until sampling semantics are reviewed.
- Decide later whether to create a recurring Codex Automation for daily triage. The current setup is ready for manual L1 runs.
